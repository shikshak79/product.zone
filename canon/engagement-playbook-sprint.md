# Product.Zone — Engagement Playbook: Full Sprint
## Buyability & Sales Motion Sprint
**Canonical Reference · v1.1 · March 2026**
*Aligned to Buying Path X-RAY Architecture v5.1*

---

## Playbook Philosophy

**The Sprint builds what the Entry Product unlocked.**

The Entry Product locked 18 positioning decisions and produced a messaging spine. The Full Sprint takes those locked decisions and renders them into a complete, aligned surface set — full homepage, pitch deck, sales brief, demo script, objection map. The founder leaves with a motion others can run.

**Same operating model as the Entry Product:**
Sessions are decision-making and review moments, not working sessions. All building work happens between sessions. The founder arrives to find drafts ready, options visible, and a recommendation on the table. Their job is to review, react, refine, and rehearse — not to produce.

**The shift from Entry Product to Sprint:**
The Entry Product is primarily analytical — decisions get locked through diagnosis and deliberation. The Sprint is primarily generative — decisions already locked, now rendered into surfaces. The founder's role shifts from choosing to rehearsing. Surfaces are iterated live. The pitch deck gets practiced and revised until it lands.

**Prerequisite without exception:**
The Full Sprint does not re-do positioning work. If the 18 decisions are not locked, the Sprint cannot start. Entry Product first.

## Engagement Overview

| | |
|---|---|
| **Product** | Buyability & Sales Motion Sprint |
| **Price** | €10,000 |
| **Sessions** | 3 working sessions (Sessions 4–6) + handover |
| **Session length** | 90–120 minutes |
| **Standard duration** | 6 weeks total (including Entry Product: weeks 4–6 of full arc) |
| **Fast track** | 4 weeks total (weeks 3–4 of compressed arc) |
| **Founder time** | ~5–6 hours (sessions + handover) |
| **Between-session work** | Observation + one explicit test per session cycle |

**Note on duration:** The 6-week standard refers to the Sprint arc in isolation. When Entry Product and Sprint are run sequentially without pause, total engagement is 6–9 weeks. When purchased after a gap following the Entry Product, the Sprint is 3–4 weeks from kickoff to handover.

## Sprint Entry Checkpoint

Before Session 4 begins, confirm the handover from the Entry Product is complete.

### Entry Checkpoint Checklist

| Condition | Must Be True |
|---|---|
| Decision ledger | All 18 decisions locked with answers in buyer language |
| Messaging spine | Complete — category, customer, alternative, difference, outcome, proof direction, first step |
| Homepage spine | Draft exists — hero + first scroll, copy-ready |
| Notion workspace | Founder has access and can navigate independently |
| Segment morphology | Target company + role confirmed |

If any condition is not met, address the gap before Session 4. Do not start surface builds on fuzzy foundations.

### Sprint Kickoff (15 min)
If entering Sprint after a pause from the Entry Product (gap of 2+ weeks), run a brief re-alignment call before Session 4:
- Recap locked decisions — anything that shifted based on real-world conversations?
- Review homepage spine — any early feedback from testing?
- Confirm calendar for Sessions 4–6 and handover
- Update decision ledger if any decisions refined since Entry Product

If continuing immediately from Entry Product, this step is not needed — proceed directly to Session 4 prep.

## Between Entry Product and Session 4: Thorsten's Prep

**Purpose:** Translate locked decisions into full homepage architecture and copy. Prepare all materials for Session 4 review.

### Prep Work Checklist

**Full homepage build:**
- [ ] Map all locked decisions to homepage sections using the decision → question mapping (v5.1 Appendix)
- [ ] Draft full homepage copy: all sections covering all 20 buyer questions (Q1–Q24)
- [ ] Score the draft against 20 buyer questions — target 18+/20 explicit before presenting
- [ ] Run PULL check on draft: Q1, Q9, Q10 must all be explicit — if not, reframe those sections
- [ ] Prepare 2–3 headline variants for key sections (founder chooses, not approves)
- [ ] Prepare Figma layout: homepage sections as visual draft (structure and copy priority)

**Pitch deck architecture:**
- [ ] Draft pitch deck structure: 10–12 slides mapped to buying journey stages
- [ ] Map each slide to its job using the 10-question deck framework (D1–D10)
- [ ] Prepare slide-by-slide outline with copy direction for each (full copy comes after Session 4 alignment)

**Session 4 content prep:**
- [ ] Homepage draft visible in Notion + Figma — annotated by decision (which decision each section answers)
- [ ] Buyer question scorecard for draft homepage — shows gaps before session
- [ ] Prepare 2–3 specific choices for founder to make in session (not open-ended reviews)

**Automation note:** Homepage copy drafting from locked decisions is highly automatable. Given a complete decision ledger and messaging spine, 80%+ of first-draft homepage copy can be AI-generated and then refined. Session 4 is the refinement and alignment session — not the writing session.

## Session 4: Buyability — Full Homepage

**Job:** Review and finalise full homepage copy. Confirm every buyer question is answered. Align on final structure before production.

**Format:** 90–120 minutes · Video call with screen share
**Notion + Figma:** Homepage draft open and live

### Session Structure

**Opening — Real-world signal check (10 min)**
- "Since the Entry Product — have you used the new messaging in any conversations? What happened?"
- "Has anyone seen the homepage spine? Any reaction?"
- Capture signal. Update decision ledger if anything refined.

**Homepage walkthrough (50 min)**
- Pull up the homepage draft in Figma/Notion
- Walk through section by section
- Run the full 20-question audit live (Q1–Q24) — score each: Explicit / Partial / Missing
- Run PULL check: are Q1 (Project), Q9 (Look), Q10 (Lacking) all explicit? If not, flag sections that are still product-centric
- Target: 18+/20 explicit before approving for production
- Make live edits in Notion where copy needs revision
- For headline variants: founder chooses, you explain the tradeoff of each

**Pitch deck structure alignment (20 min)**
- Present the pitch deck outline (slide-by-slide structure)
- Align on: which decisions does each slide carry?
- Confirm the story arc: does the deck walk the buyer from Land → Compare → Validate → Commit?
- Flag any slides that will require founder-specific input (customer stories, specific proof moments)

**Close (10 min)**
- Confirm: homepage is approved for production (or note remaining revisions)
- Preview Session 5: "We'll build and rehearse the full pitch deck"
- Set explicit test for between sessions: "Use the new homepage language in your next two sales conversations. Notice which phrases get the strongest reaction — we'll use that signal in Session 5."

### Session 4 Output (Thorsten delivers within 24–48 hours)
- Homepage copy finalised (all sections, production-ready — answers 20 buyer questions)
- Homepage buyer question scorecard (final score __/20 explicit)
- Pitch deck structure confirmed (slide-by-slide outline with D1–D10 question mapping)
- Any homepage revisions from session incorporated

## Between Session 4 and Session 5: Thorsten's Prep

**Purpose:** Build the full pitch deck. Prepare demo script framework.

### Prep Work Checklist

**Pitch deck build:**
- [ ] Write full pitch deck copy — all 10–12 slides, based on confirmed structure and locked decisions
- [ ] Map each slide to its job in the buying journey (Land / Make Sense / Compare / Validate / Commit)
- [ ] Build deck in Figma using Product.Zone deck template + client visual system
- [ ] Prepare slide-by-slide speaker notes (what founder says on each slide — repeatable arc)
- [ ] Identify slides requiring founder customisation (specific proof, specific customer story)

**Demo script draft:**
- [ ] Draft 5-minute demo arc mapped to the 7-question demo framework (M1–M7):
  - M1: What will I see in the next 5 minutes? (Category, Outcome — preview stated)
  - M2: How is this different from X? (Alternative — named + defeated)
  - M3: Why should I care? (Difference — mechanism connected to pain)
  - M4: What result does this create? (Outcome — measurable result shown)
  - M5: Does this work for teams like mine? (Proof — relevant proof surfaced)
  - M6: What's the risk? (Primary Risk — named + mitigated)
  - M7: What happens if I want to proceed? (First Step, Start Recipe — clear next step)
- [ ] Prepare speaking guide: exact phrasing for each segment (repeatable by anyone)

**Objection preparation:**
- [ ] List top 5–7 objections likely to surface in pitch and demo
- [ ] Map each objection to buying stage (Compare, Validate, Commit)
- [ ] Prepare initial asset map: objection → what to send/say → which Notion/Figma asset supports

**Session 5 content prep:**
- [ ] Pitch deck complete in Figma — ready to present and rehearse
- [ ] Speaker notes complete for all slides
- [ ] Demo script printed / visible in Notion
- [ ] Identify 3 slides most likely to need revision based on founder context

**Founder observation (implicit):**
Founder used new homepage language in 1–2 conversations. Any signal from that feeds into Session 5 slide copy and objection preparation.

## Session 5: Sales Pitch — Build and Rehearse

**Job:** Present the complete pitch deck. Rehearse the delivery. Revise until it lands. Confirm demo script structure.

**Format:** 90–120 minutes · Video call with screen share
**Figma:** Pitch deck open and live

### Session Structure

**Opening — Conversation signal debrief (10 min)**
- "Did you use the new framing in any conversations since Session 4? What phrases landed? What got a reaction?"
- Capture signal. Update any slides based on what worked in practice.

**Pitch deck walkthrough — first pass (25 min)**
- Walk through every slide
- For each: does this do its job? Does it answer the right buyer question at this stage?
- Founder reactions: what feels right, what feels off, what's missing
- Mark revision priorities

**Live rehearsal (30 min)**
- Founder presents the deck as if to a prospect — you play the prospect
- Do not interrupt during the run
- After: give specific feedback on phrasing, pacing, which slides cause hesitation
- Revise 2–3 key slides based on rehearsal
- Optional second run if major revisions made

**Travelability check (10 min)**
After rehearsal, run the formal Travelability checklist — can the champion carry this without the founder?
- [ ] Can buyer retell this to their boss in one sentence?
- [ ] Is the problem statement specific and memorable (not generic or jargon-heavy)?
- [ ] Can champion answer "why this vs alternatives?" (Alternative named + Difference clear?)
- [ ] Does the proof match this buyer's context (ICP-matched case study, not generic logos)?
- [ ] Is the ask clear and safe (specific next step, low commitment)?

**Score: __/5 pass.** If <3 pass → deck requires founder on every call. Travelability is RED — revise before next real use.
- Walk through 5-minute demo arc
- Is the hook strong enough? Does the problem section name the alternative?
- Does the show segment demonstrate the locked outcome, not just features?
- Founder practises the hook and problem section aloud
- Confirm structure — full demo script refinement comes in Session 6

**Close (10 min)**
- Preview Session 6: "We build the sales brief, finalise the demo script, and complete the objection map"
- Explicit test: "Run this deck in your next real conversation. What question comes up that we haven't pre-handled? Bring that to Session 6."

### Session 5 Output (Thorsten delivers within 24–48 hours)
- Pitch deck revised from session (Figma — production-ready)
- Speaker notes updated
- Demo script draft (structure + phrasing for all 5 segments)
- Session notes: which objections came up in rehearsal

## Between Session 5 and Session 6: Thorsten's Prep

**Purpose:** Build the complete sales motion handoff package — sales brief, finalised demo script, objection map.

### Prep Work Checklist

**Sales brief:**
- [ ] Draft Day-1 Sales Brief: one-pager a rep or agency can use without founder in the room
- [ ] Must include: one-liner, ICP definition, alternative contrast, proof points, first step offer, objection responses (top 5)
- [ ] Format: Notion page + PDF export

**Demo script finalisation:**
- [ ] Finalise 5-minute demo script with exact phrasing for all segments
- [ ] Add: what to show (specific screen or moment), what to say, what reaction to listen for
- [ ] Format: Notion page with timed arc + speaker guide

**Objection → asset map:**
- [ ] Complete list of top 7–10 objections across all buying stages (Compare, Validate, Commit)
- [ ] For each objection: what the buyer is really asking, what to say, which asset to send
- [ ] Assets referenced must exist in the Notion workspace (proof points, homepage sections, deck slides)

**Buying path documentation:**
- [ ] Document how the buyer moves from Land → Commit using the locked decisions and surfaces
- [ ] Map each surface to its role in the buying journey
- [ ] Note: what triggers progression at each stage?

**Handover package prep:**
- [ ] Audit entire Notion workspace — is everything complete, organised, navigable?
- [ ] Ensure all Figma assets linked correctly
- [ ] Prepare handover session agenda

## Session 6: Sales Motion — Install and Handover Prep

**Job:** Review sales brief and demo script. Complete objection map. Confirm the full motion is installed and founder can brief anyone from it.

**Format:** 90–120 minutes · Video call with screen share
**Notion:** Sales brief, demo script, objection map all open

### Session Structure

**Opening — Pitch deck field test debrief (10 min)**
- "Did you run the deck in a real conversation? What question came up that we haven't handled?"
- Capture objection. Add to objection map live in Notion.
- Note any copy adjustments triggered by real use.

**Sales brief review (20 min)**
- Walk through Day-1 Sales Brief
- Test: "Could a new rep read this and know what to say in the first 5 minutes of a call?"
- Revise any section that requires founder explanation to understand
- Confirm: one-liner, ICP definition, proof points, objection responses all sharp

**Demo script walkthrough and rehearsal (25 min)**
- Walk through the finalised demo script segment by segment (M1–M7)
- M1: Does the preview hook immediately? Does it name the outcome upfront?
- M2: Is the alternative named explicitly? Does it get defeated, not just mentioned?
- M3: Is the mechanism connected to the named pain — not just a feature demo?
- M4: Is the result measurable, not aspirational?
- M5: Is the proof ICP-matched — same company type, same role?
- M6: Is the risk named and mitigated — not ignored?
- M7: Is the next step specific and low-commitment?
- Founder rehearses M1 and M2 aloud
- Test: anyone who reads this script can run the demo — no tribal knowledge required

**Objection map review (20 min)**
- Walk through all 7–10 objections
- For each: is the response clear? Does it address what the buyer is really asking? Is the asset reference correct?
- Add any objections surfaced in the field test
- Confirm the map covers all buying stages

**Full motion overview (15 min)**
- Pull back: walk through the complete surface set as a buyer would experience it
  - Finds you (LinkedIn / outbound) → Homepage → Pitch deck → Demo → Sales brief (champion shares) → Closes
- Confirm: each surface does its job, each transition is clean
- Note: what should the founder test in the first 30 days with this motion?

### Session 6 Output (Thorsten delivers within 48 hours)
- Sales brief finalised (Notion page + PDF)
- Demo script finalised (Notion page with timed arc)
- Objection → asset map complete
- Buying path documentation
- Full workspace audit complete — all assets in place

## Handover Session

**Format:** 60 minutes · Video call
**Timing:** 3–5 days after Session 6

### Agenda

**Complete workspace walkthrough (30 min)**
Walk through every deliverable in the Notion workspace:

| Deliverable | Where | How to use |
|---|---|---|
| Buying path diagnostic report | Notion | Reference for ongoing self-diagnosis |
| Decision ledger (18 decisions) | Notion | Source of truth for all surfaces |
| Messaging spine | Notion | Copy foundation for any new surface |
| Homepage copy | Notion + Figma | Brief to developer / designer |
| Pitch deck | Figma | Present via deck link (trackable) |
| Sales brief | Notion + PDF | Day-1 brief for rep or agency |
| Demo script | Notion | Train anyone on product demo |
| Objection → asset map | Notion | Sales enablement reference |
| Buying path documentation | Notion | Onboarding for first sales hire |
| Asset library | Notion | All Figma links, version-controlled |

**Usage guidance (15 min)**
- How to maintain the decision ledger as positioning evolves
- How to use the messaging spine when briefing content, ads, or new surfaces
- How to onboard a first sales rep using the sales brief and demo script
- What to track in the first 30 days (leading indicators from the Buying Path Framework)

**What to test first (10 min)**
- Agree on 2–3 leading indicators to watch in the first 4 weeks
- Set a check-in point (typically 3–4 weeks) — this is the natural Stabilisation Retainer conversation
- If Stabilisation Retainer makes sense, confirm format and start date

**Close (5 min)**
- Confirm all access (Notion, Figma)
- Confirm export formats for any deliverables (PDF, PNG, etc.)
- Confirm async channel for any questions in the first week

## Quality Standards

**Every session must end with:**
- At least one surface approved or a clear revision list
- Founder has practised something aloud (pitch, hook, or demo segment)
- Next session agenda previewed

**Every build block must produce:**
- First draft complete before session — founder never sees a blank page in a session
- 2–3 variants for key choices — founder chooses, does not just approve
- Annotation explaining why each choice was made (decision-level rationale, not aesthetic)

**Handover is complete when:**
- A first sales rep could be onboarded using only the Notion workspace
- All surfaces are aligned — same story, same language, same decisions
- Founder knows what to measure in the first 30 days and how
- Every deliverable has a named use case (who uses it, when, for what)

## The Full Arc (Entry Product + Full Sprint)

When both products are run sequentially, the complete engagement looks like this:

```
WEEKS 1–3 (STANDARD) / WEEKS 1–2 (FAST TRACK)
  Entry Product: Positioning & Messaging Sprint
  Sessions 1–3 + Handover
  Output: 18 decisions locked, messaging spine, homepage spine

WEEKS 4–6 (STANDARD) / WEEKS 3–4 (FAST TRACK)
  Full Sprint: Buyability & Sales Motion Sprint
  Sessions 4–6 + Handover
  Output: Full surface set, complete motion, handoff-ready

WEEKS 7–12 (OPTIONAL)
  Stabilisation Retainer
  Bi-weekly sessions + async
  Output: Positioning stable, next Sprint hypothesis ready
```

**Total founder time for full arc:** ~11–13 hours across 6–9 weeks.
**Total elapsed time (standard):** 9 weeks including handovers.
**Total elapsed time (fast track):** 6 weeks including handovers.
